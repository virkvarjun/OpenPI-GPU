## ⚠️ Measurement note — the 95.5% "data-wait" is an ARTIFACT here, not a signal

This harness re-runs a **fixed, already-on-device input** to attribute the compiled step's device ops. It has
**no data loader and no host→device transfer**, so "data-wait" (`wall − device_busy`) here is **host dispatch +
profiler overhead**, not input starvation. It's large only because the GPU step is so fast (~11 ms) that
per-step Python dispatch dominates wall time; in a real training loop this overhead is amortized/overlapped
across pipelined steps. **Do not read 95.5% as "the model is input-bound."**

## What IS trustworthy: the device-compute composition

Of the ~10.9 ms the GPU is actually busy per forward+backward step:
- **matmul-FFN dominates (~65%)** — the dense GEMMs (gemma + SigLIP projections, FFN, and the attention QK/AV
  dots). Per the audit + decision gate, **XLA already saturates these — do not touch them.**
- **attention (softmax) is ~0%** — at batch 1 / short sequence the naive masked softmax is negligible next to
  the GEMMs. **⇒ No fused-attention headroom at this config** (the "attention > 15–20%" gate is NOT triggered).
- vision/embedding are small; `other` (~31% of compute) is RNG (dropout/aug), norms, casts, flow-matching ops.

## Routing decision

- ❌ **Not** fused attention — attention is ~0% of compute.
- ❌ **Not** matmul/GEMM — saturated by XLA (audit rule).
- ❓ **data-wait / input pipeline** — the *likely* real lever for an image-heavy VLA, BUT this microbenchmark
  cannot quantify it (no data pipeline). **Next measurement required:** attribute the **real `train.py`
  `ptrain_step` loop with the actual data loader** over many pipelined steps, where data-wait is genuine and
  dispatch is amortized. Only then is the input-pipeline optimization justified by measurement.

**Conclusion:** the model compute is GEMM-bound with no attention headroom; the next step is to measure the
*real* training loop (data loader feeding the GPU) for a trustworthy data-wait number before optimizing the
input pipeline. The attribution instrument + GPU harness are validated and ready for that loop-level run.
